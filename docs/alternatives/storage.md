# 로컬 저장소 대안 분석

> 드로잉 데이터의 빈번한 자동 저장과 메타데이터 쿼리를 모두 만족하는 저장소 전략을 비교한다.

---

## 1. 데이터베이스 / Key-Value 저장소

### 요약 비교

| 솔루션 | AsyncStorage 대비 속도 | SQL 쿼리 | 동기 API | 암호화 | 클라우드 호환 |
|--------|----------------------|----------|----------|--------|-------------|
| **expo-sqlite** | 1~2x | ★★★★★ | ○ (JSI) | ✗ | 우수 |
| **MMKV** | **30x** | ✗ | ★★★★★ | ○ | 수동 |
| **WatermelonDB** | 4x | ORM 스타일 | ✗ | ✗ | 우수 |
| **Realm** | 빠름 | 자체 쿼리 API | ○ | ○ | 내장 동기화 |
| AsyncStorage | 1x (기준) | ✗ | ✗ | ✗ | 수동 |

---

### expo-sqlite (현재 선정)

```
PostgreSQL 경험이 있는 승원님에게 가장 자연스러운 선택.
SQL 문법이 거의 동일하고, 테이블 설계 방식도 유사.
```

**장점**
- 완전한 SQL 지원 → PostgreSQL 경험 직접 활용
- `expo-sqlite/next`에서 JSI 기반 동기 호출 지원 (성능 개선)
- Expo 관리 환경에서 가장 안정적
- 주간 다운로드 24만+ (높은 채택률)
- Supabase(PostgreSQL) 동기화 시 스키마 매핑이 자연스러움

**단점**
- Key-Value 저장에는 과도한 구조 (MMKV 대비)
- 빈번한 쓰기(매 stroke 자동저장)에 최적은 아님
- 트랜잭션 오버헤드

---

### MMKV (고빈도 쓰기 최적)

```
Spring으로 비유하면: Redis 캐시.
구조화된 쿼리는 불가하지만 읽기/쓰기 속도가 압도적.
```

**장점**
- **AsyncStorage 대비 30x 빠른 읽기/쓰기**
- 동기(synchronous) API → async/await 불필요
- 암호화 지원
- 매우 가벼움

**단점**
- SQL 쿼리 불가 → "3월의 모든 글" 같은 조회 구현이 어려움
- Key-Value 구조 → 복잡한 관계 표현 한계
- 데이터 마이그레이션 도구 부재

**핵심 사용처**: 캔버스 자동저장 (작성 중인 stroke 데이터의 빈번한 임시 저장)

---

### WatermelonDB (복잡한 관계형 데이터)

```
Spring으로 비유하면: JPA + Hibernate.
모델 정의 → 자동 쿼리 생성 → lazy loading까지 유사한 패턴.
```

**장점**
- AsyncStorage 대비 4x 빠름
- 쿼리가 별도 네이티브 스레드에서 실행 → UI 블로킹 없음
- ORM 스타일 모델 정의로 관계 표현 용이
- 클라우드 동기화 아키텍처가 잘 설계되어 있음

**단점**
- 초기 설정이 expo-sqlite 대비 복잡
- Expo와의 통합에 추가 설정 필요
- 프로토타입에 과도한 구조일 수 있음

---

### Realm

**장점**
- 실시간 동기화 내장 (MongoDB Atlas와 연동)
- Zero-copy 읽기로 빠른 응답
- 자체 쿼리 API 제공

**단점**
- MongoDB 생태계 종속
- 일부 리서치에서 2024년 이후 지원 축소 언급
- PostgreSQL 기반 클라우드(Supabase)와 호환 불리

---

### 권장: 하이브리드 전략 (MMKV + expo-sqlite)

```mermaid
graph LR
    Canvas["캔버스 필기 중"] -->|매 stroke| MMKV["MMKV<br/>(임시 저장, 30x 빠름)"]
    MMKV -->|저장 버튼| SQLite["expo-sqlite<br/>(메타데이터 + 최종 저장)"]
    SQLite -->|후속 단계| Supabase["Supabase<br/>(클라우드 동기화)"]
```

| 역할 | 저장소 | 근거 |
|------|--------|------|
| 캔버스 자동저장 (임시) | MMKV | 고빈도 쓰기 최적, 동기 API로 UI 블로킹 없음 |
| 일기 메타데이터 (제목, 날짜, 태그) | expo-sqlite | SQL 쿼리로 검색/정렬/필터링 |
| 완성된 드로잉 파일 | expo-file-system | 바이너리/JSON 파일 저장 |

**단순 접근 대안**: 프로토타입에서 MMKV 없이 expo-sqlite만 사용하는 것도 가능. 자동저장 빈도를 조절하면(예: 1초 디바운스) sqlite로도 충분할 수 있음. 복잡도를 줄이는 것이 프로토타입에서는 더 중요할 수 있다.

## 2. 파일 저장소

| 솔루션 | 용도 | 주간 다운로드 | Expo 호환 | 비고 |
|--------|------|--------------|-----------|------|
| **expo-file-system** | Expo 프로젝트 표준 | 190만+ | ★★★★★ | **권장** |
| react-native-blob-util | 대용량 파일, 멀티파트 업로드 | 27만+ | ★★★☆☆ | 스트리밍 필요 시 |
| react-native-fs | 레거시 | 19만+ | ★★☆☆☆ | Expo에서 비권장 |

**결론**: Expo 기반이므로 `expo-file-system`이 자연스러운 선택. 클라우드 업로드 시 스트리밍이 필요하면 `react-native-blob-util`을 보완적으로 사용.

## 3. 클라우드 저장소 (후속 단계)

| 서비스 | 무료 스토리지 | 이그레스 비용 | PostgreSQL 포함 | RN 통합 |
|--------|-------------|-------------|-----------------|---------|
| **Supabase** | 1GB 파일 + 500MB DB | 포함 | ★★★★★ | 공식 SDK |
| Firebase Storage | 5GB | 월 $50 크레딧 | ✗ | 우수 |
| Cloudflare R2 | 10GB | **무료** | ✗ | S3 호환 API |
| AWS S3 | 5GB (12개월) | $0.09/GB | ✗ | 풍부 |

### Supabase (권장)

**장점**
- PostgreSQL 기반 → expo-sqlite 스키마를 거의 그대로 마이그레이션
- Row Level Security로 사용자 데이터 격리
- React Native 공식 SDK + Expo 가이드 제공
- 무료 티어로 프로토타입 충분

**단점**
- 무료 프로젝트가 **1주 미사용 시 자동 일시 중지** (최대 2개 활성)
- 파일 스토리지 1GB는 이미지 기반 앱에서 빠르게 소진 가능

### Cloudflare R2 (다운로드 트래픽 많을 때)

**장점**
- **이그레스 비용 무료** → 커뮤니티 기능에서 이미지 다운로드 비용 절감
- 10GB 무료 스토리지
- S3 호환 API

**단점**
- 데이터베이스 미포함 → Supabase DB와 병행 필요
- React Native 전용 SDK 없음 (S3 클라이언트로 접근)

### 권장 조합 (후속 단계)

```
메타데이터: Supabase PostgreSQL
드로잉 파일: Supabase Storage (초기) → Cloudflare R2 (트래픽 증가 시)
```
