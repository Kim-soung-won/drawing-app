# 상태 관리 대안 분석

> 캔버스의 실시간 드로잉 상태와 Undo/Redo 히스토리를 다루기에 적합한 상태 관리 라이브러리를 비교한다.

---

## 요약 비교

| 라이브러리 | 번들 크기 | 1000회 업데이트 | 러닝커브 | Undo/Redo 적합도 | 비고 |
|-----------|-----------|-----------------|----------|------------------|------|
| **Zustand** | ~3KB | 12ms | 매우 낮음 | ★★★★☆ | **현재 선정** |
| Redux Toolkit | ~15KB | 18ms | 중간 | ★★★★★ | 시간여행 디버깅 내장 |
| Jotai | ~2.5KB | 14ms | 낮음~중간 | ★★★★☆ | 원자적 상태 모델 |
| Legend-State | ~2KB | 최상* | 낮음~중간 | ★★★★★ | 시그널 기반, 동기화 내장 |
| MobX | ~15KB+ | 우수 | 중간~높음 | ★★★★☆ | 자동 반응성 |
| React Context | 0KB | 낮음 | 매우 낮음 | ★★☆☆☆ | 리렌더링 이슈 |

*Legend-State: 일부 벤치마크에서 바닐라 JS보다 빠른 결과

## 상세 분석

### 1. Zustand (현재 선정)

```
Spring으로 비유하면: @Component로 등록한 싱글톤 빈에 직접 접근하는 느낌.
별도 프레임워크 없이 간결하게 상태를 선언하고 사용.
```

**장점**
- 보일러플레이트 최소: store 하나로 완결
- React 외부에서도 상태 접근 가능 (`getState()`)
- 선택적 구독으로 불필요한 리렌더링 방지
- Tree-shakeable → 사용한 것만 번들에 포함

**단점**
- 시간여행 디버깅이 내장되지 않음 (미들웨어로 구현 가능하지만 별도 작업)
- 대규모 팀에서 컨벤션 강제력 부족 (자유도가 높은 만큼)
- DevTools가 Redux 대비 빈약

**Undo/Redo 구현 패턴**
```typescript
interface CanvasState {
  past: Stroke[][];
  present: Stroke[];
  future: Stroke[][];
  addStroke: (stroke: Stroke) => void;
  undo: () => void;
  redo: () => void;
}
```

---

### 2. Redux Toolkit

```
Spring으로 비유하면: Spring MVC의 엄격한 레이어 구조.
Action → Reducer → State의 단방향 흐름이 Controller → Service → Repository 패턴과 유사.
```

**장점**
- **시간여행 디버깅 내장** → Undo/Redo와 자연스러운 궁합
- Redux DevTools로 모든 상태 변화 추적 가능
- 미들웨어 시스템으로 사이드이펙트 관리 용이
- 팀 규모가 커져도 예측 가능한 상태 흐름

**단점**
- 번들 크기 ~15KB (Zustand의 5배)
- 보일러플레이트 여전히 존재 (RTK가 줄였지만 Zustand 대비 많음)
- 프로토타입에 과도한 구조

**트레이드오프**: Undo/Redo 구현의 편의성은 Redux가 우세하지만, 프로토타입에서의 개발 속도는 Zustand가 앞선다. 혼자 개발하는 프로토타입에서 Redux의 엄격한 구조는 오버헤드가 될 수 있음.

---

### 3. Jotai

```
Spring으로 비유하면: @Value로 개별 속성을 주입받듯,
상태를 원자(atom) 단위로 쪼개어 필요한 곳에서만 구독.
```

**장점**
- 번들 크기 최소 (~2.5KB)
- 원자적 상태 → 캔버스의 개별 설정(펜 색상, 굵기 등)을 독립적으로 관리 가능
- 불필요한 리렌더링 자연스럽게 방지
- React Suspense와 자연스럽게 통합

**단점**
- 상태 간 의존 관계가 복잡해지면 atom 그래프 파악이 어려움
- DevTools 지원이 Zustand/Redux 대비 부족
- 전역 상태보다 컴포넌트 레벨 상태에 더 적합

**트레이드오프**: 캔버스 도구 설정(펜, 색상, 굵기)처럼 독립적인 상태에는 Jotai가 우아하지만, Undo/Redo처럼 여러 상태를 묶어 스냅샷으로 관리해야 하는 경우에는 Zustand/Redux의 단일 스토어가 더 직관적.

---

### 4. Legend-State

**장점**
- 시그널 기반 → 가장 세밀한 리렌더링 제어
- **동기화 엔진 내장**: 로컬 → 클라우드 동기화를 라이브러리 레벨에서 지원
- 번들 크기 ~2KB로 최소
- 일부 벤치마크에서 바닐라 JS보다 빠른 성능

**단점**
- React 생태계에서 상대적으로 신생
- 커뮤니티 규모가 작아 트러블슈팅 자료 부족
- 시그널 패턴이 React의 일반적인 멘탈 모델과 다름

**트레이드오프**: 후속 단계에서 클라우드 동기화를 구현할 때 Legend-State의 내장 동기화가 큰 이점이 될 수 있다. 하지만 프로토타입에서는 커뮤니티 자료가 풍부한 Zustand이 트러블슈팅에 유리.

---

### 5. React Context (비권장)

**왜 적합하지 않은가**
- Context 값이 변경되면 **모든 Consumer가 리렌더링**
- 캔버스에서 매 stroke마다 Context가 업데이트되면 전체 UI가 리렌더링
- Context 분리(데이터 vs 액션)로 완화 가능하나 복잡도만 증가
- 상태 관리 라이브러리가 해결해주는 문제를 수동으로 관리해야 함

**유일한 사용처**: 테마, 로케일 등 거의 변하지 않는 설정값에 한정.

| 단계 | 권장 | 이유 |
|------|------|------|
| **프로토타입** | Zustand | 최소 보일러플레이트, 빠른 구현, 충분한 성능 |
| **클라우드 동기화 추가 시** | Legend-State 검토 | 내장 동기화 엔진이 아키텍처를 단순화 |
| **팀 확장 시** | Redux Toolkit 검토 | 엄격한 패턴으로 협업 예측성 향상 |
